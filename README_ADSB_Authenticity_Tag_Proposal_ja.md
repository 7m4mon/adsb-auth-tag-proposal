# Simple ADS-B Authenticity Tag Proposal

日付: 2026年5月11日  
Author: 7M4MON

## 概要

これは、ADS-B のなりすまし対策について考えた、追加認証フレームの提案です。

ADS-B は、航空機が自機の位置・高度・速度・識別情報などを周囲の航空機や地上局にブロードキャストする仕組みです。しかし、現在の ADS-B メッセージ自体には、暗号学的な認証機構がありません。

そのため、技術的には ADS-B らしい信号を送信する装置を作ることは比較的容易であり、悪意ある送信機によって「存在しない航空機」や「別の場所にいるように見える航空機」を作り出せてしまう可能性があります。

これは困ります。

本提案の目的は、軽量な仕組みによって、少なくとも次のような判断を支援することです。

> 「少なくとも、その時刻に、その正規の機体が、その空域にいた」

この文書では、そのための追加認証フレームのアイデアを説明します。

---

## 背景

ADS-B のなりすまし対策については、すでに本格的な提案があります。

例：

- TESLA を用いた ADS-B 認証
- PKI を用いた航空機証明書
- Phase Overlay を用いた追加データ伝送
- CABBA: Compatible Authenticated Bandwidth-efficient Broadcast protocol for ADS-B
- Secure Authentication of ADS-B Aircraft Communications using Retroactive Key Publication
- IETF draft: ADS-B Authentication

一方で、このリポジトリで扱う案は、もっと基礎的な理論を元にしたものです。

---

## 基本アイデア

ADS-B に新しいフレーム種類を 1 つ追加し、そこに短い認証値を載せます。

既存の位置情報フレーム、速度フレーム、Aircraft Identification フレームなどは変更しません。

互換性を持たせるため、追加する認証フレームは、現在の ADS-B Extended Squitter フレームと同じ全体構造を使います。

```text
DF17 ADS-B Extended Squitter

| DF | CA | ICAO Address | ME Field | PI |
```

このうち、ME フィールドの Type Code に予約領域の値を仮に割り当て、認証タグ用メッセージとして扱います。

---

## 仮のフレーム割当

この提案では、DF や CA を新たに定義するのではなく、既存の 1090ES ADS-B フレームの枠組みを利用します。

仮の割当は以下とします。

| Field | Value |
|---|---:|
| DF | 17 |
| CA | 既存の transponder capability 値を使用 |
| ME Type Code | 24 |
| PI | 通常の Mode S parity |

Type Code 23〜27 は Reserved とされているため、本提案では仮に Type Code 24 を `Proposed Authenticity Tag` として使用します。

ただし、ME フィールド全体は 56 bit ですが、その先頭 5 bit は Type Code として使用されます。  
そのため、timestamp と HMAC に使える領域は 51 bit です。

本提案では以下の割当とします。

| Field | bits | 内容 |
|---|---:|---|
| Type Code | 5 | 24: Proposed Authenticity Tag |
| Timestamp | 27 | 1秒単位、約4.25年周期 |
| Truncated HMAC | 24 | 単発偽造成功率 1 / 16,777,216 |

```text
ME field: 56 bits

+-----------+---------------------------+------------------------+
| Type Code | Timestamp                 | Truncated HMAC         |
| 5 bits    | 27 bits                   | 24 bits                |
+-----------+---------------------------+------------------------+
```

---

## HMAC の計算対象

HMAC は、機体または送信機ごとの共有秘密鍵を使って計算します。

例：

```text
HMAC = Truncate24(HMAC-SHA256(secret_key, message))
```

MAC 対象は以下のとおりです。

```text
DF
CA
ICAO address
ME Type Code
Timestamp
```

ここで使っている HMAC は、共有鍵による認証方式です。公開鍵暗号による電子署名ではありません。

---

## この方式で何を証明したいか

この方式で証明したいのは、次のことです。

> その時刻に、その共有秘密鍵を持つ正規の送信機が、この認証タグを生成した

ADS-B フレーム上に載せるのは、24 bit の短い認証タグだけです。そのため、検証は、正規送信機と対応する共有秘密鍵を持つインターネット上の検証サーバーで行うことを想定します。

受信側は、受信した ADS-B 認証フレームを、たとえば HEX 文字列として検証サーバーに送信します。検証サーバーは、ICAO アドレスに対応する共有秘密鍵を使って truncated HMAC を再計算し、受信フレーム内の tag と一致するか確認します。

これにより、次の判断の信頼度を高めることができます。

> その鍵に対応する航空機または送信機が、その時刻に有効な認証タグを生成した

ただし、この方式だけで物理的な位置そのものを証明できるわけではありません。位置の妥当性は、受信信号の観測、MLAT、複数受信局での整合性、その他の独立した方法で確認する必要があります。

---

## インターネット接続と検証サーバー

この方式では、受信側がオンラインの検証サーバーと通信できることを想定します。

検証サーバーは、以下のような情報を管理または参照します。

- ICAO address
- Key ID
- 共有 HMAC 秘密鍵、または保護された鍵情報
- 鍵の有効期間
- 鍵更新状態
- 失効情報

つまり、完全にオフラインの受信機だけで完結する方式ではありません。

インターネットに接続された地上受信局や、ネットワーク接続された ADS-B 受信システムでの検証を想定します。

受信機は共有秘密鍵そのものを知る必要はありません。代わりに、受信したフレームを検証サーバーに送り、次のような結果を受け取ります。

```json
{
  "ok": true,
  "valid_crc": true,
  "valid_auth": true,
  "icao": "ABC123"
}
```

## なぜ 27bit Timestamp + 24bit HMAC なのか

27bit timestamp は、1秒単位で約4.25年持ちます。

```text
2^27 seconds = 134,217,728 seconds
134,217,728 / 60 / 60 / 24 / 365 ≒ 4.25 years
```

24bit HMAC の単発偽造成功率は以下です。

```text
1 / 2^24 = 1 / 16,777,216
```

24bit HMAC は暗号学的には短いです。しかし、この方式は本格的な暗号認証を目的としたものではなく、既存 ADS-B との後方互換性を保った軽量ななりすまし抑止を目的としています。

ADS-B の場合、HMAC を総当たりするには、攻撃者は 1090 MHz で大量の偽 ADS-B フレームを送信する必要があります。

これは実質的に、1090 MHz 帯を妨害または汚染している状態に近くなります。

そのような大量試行は、以下のような方法で検出できると思われます。

- 同一 ICAO アドレスに対する大量の認証失敗
- 異常なフレーム密度
- 複数受信局での不自然な受信状況

HMAC を長くすることも重要ですが、現実的な脅威としては、古い正規メッセージのリプレイ攻撃の方が厄介と考えます。

---

## 4.25年後のリプレイ問題と鍵更新

27bit timestamp は約4.25年で一周します。

そのため、理論上は、次のようなリプレイ攻撃が考えられます。

> 約4.25年前に録音した正規メッセージを、同じ timestamp になるタイミングで再送する

この対策として、機体ごとの HMAC 鍵を timestamp 周期より短い間隔で更新します。

例えば、1年ごと、またはフライトごとの鍵更新を行えば、過去周期のメッセージを再送しても、現在の鍵では HMAC 検証に失敗します。

よって、以下の条件を満たすべきです。

```text
Key update interval < Timestamp cycle
```

この観点では、24bit timestamp ＝ 約194日周期、よりも 27bit timestamp の方が扱いやすいと考えます。

---

## 鍵の漏洩と更新

この方式では、機体または送信機ごとに共有秘密鍵を持つことを想定します。

もし秘密鍵が盗まれた場合、その鍵を用いた HMAC は信頼できなくなります。

その場合は、以下の対応が必要です。

- 秘密鍵の失効
- HMAC 鍵の更新
- 検証サーバーの鍵データベース更新
- 失効情報の配布または同期
- 漏洩した鍵で生成されたフレームの拒否

つまり、鍵管理の仕組みは必須です。

この提案では、実運用における鍵管理システムの詳細は定義しません。実際の運用では、鍵の発行、保管、更新、失効を安全なプロセスで行う必要があります。

---

## 既存 ADS-B との互換性

この案では、既存の ADS-B メッセージは変更しません。

通常の ADS-B 受信機は、追加された認証フレームを理解できないかもしれませんが、既存の位置情報、速度、識別情報フレームは従来どおり受信できます。

したがって、後方互換性は維持されます。

```text
既存受信機:
  通常の ADS-B フレームだけを見る

対応受信機:
  通常の ADS-B フレームに加えて、追加の認証フレームも見る
  受信した認証フレームをオンライン検証サーバーに送る
  検証サーバーから認証結果を受け取る
```

---

## 「前のメッセージに HMAC をかける」案

認証対象の情報量を増やすために、次のような案も考えられます。

```text
Authentication frame = HMAC of the immediately preceding ADS-B message
```

しかし、ADS-B は一方向ブロードキャストであり、ACK も再送もありません。

そのため、受信側が必ず直前のメッセージを受信できているとは限りません。

シーケンス番号を入れることも考えられますが、既存フレームとの互換性が失われます。

## 実証用プログラム

このリポジトリには、基本アイデアを試すための簡単な Python プログラムを含めます。

これらのプログラムは、HEX 文字列の生成、デコード、検証を行うためのものです。RF 送信用のツールではありません。

| File | Purpose |
|---|---|
| `generate_key.py` | パスワードまたはパスフレーズから HMAC 共有秘密鍵ファイルを生成する |
| `make_auth_frame.py` | DF17 TC24 認証タグフレームを 28文字の HEX 文字列として生成する |
| `decode_auth_frame.py` | 112bit HEX フレームをデコードし、Mode S CRC/parity を確認する |
| `verify_server.py` | 受信フレームを共有秘密鍵で検証する HTTP サーバーを起動する |

### 実行例

鍵ファイルを生成します。

```bash
mkdir keys
python generate_key.py --icao ABC123 --password testpass --out keys/key_ABC123.json
```

認証フレームを生成します。

```bash
python make_auth_frame.py --key keys/key_ABC123.json --ca 5 --verbose
```

生成されたフレームをデコードします。

```bash
python decode_auth_frame.py 8DABC123C0XXXXXXXXXXXXXX
```

検証サーバーを起動します。

```bash
python verify_server.py --key-dir keys --host 127.0.0.1 --port 8000
```

HTTP 経由でフレームを検証します。

```bash
curl "http://127.0.0.1:8000/verify?frame=8DABC123C0XXXXXXXXXXXXXX"
```

古いテストフレームを検証したい場合は、timestamp の鮮度確認を無効化できます。

```bash
python verify_server.py --key-dir keys --max-age -1
```

### PoC の検証フロー

```text
make_auth_frame.py
  DF / CA / ICAO / Type Code / Timestamp
  + shared secret key
  -> 24-bit truncated HMAC
  -> DF17 TC24 frame HEX

decode_auth_frame.py
  frame HEX
  -> DF / CA / ICAO / ME / PI
  -> CRC/PI check
  -> timestamp and Tag24 extraction

verify_server.py
  frame HEX
  -> CRC/PI check
  -> ICAO key lookup
  -> HMAC recomputation
  -> Tag24 comparison
  -> JSON result
```

### 重要な安全上の注意

生成したフレームを 航空用周波数 で送信しないでください。

このリポジトリに含まれるスクリプトは、オフラインのシミュレーション、プロトコル検討、管理された実験環境でのデモを目的としたものです。

## この方式でできること

- 単純な ADS-B なりすましの抑止
- 適切な鍵更新を行った場合の古いリプレイ攻撃の抑止
- 共有秘密鍵を持つ主体が認証タグを生成したことの確認
- インターネット接続された検証サーバーによる真正性確認
- 既存 ADS-B 受信機との後方互換性維持

---

## 制限事項

この提案は、重要な制限があります。

- 24bit tag は短く、本格的な暗号認証強度はありません。
- 検証サーバーは共有秘密鍵を安全に保護する必要があります。
- 受信側はオンライン検証のためにインターネット接続が必要です。
- 有効な認証タグは、航空機の物理的な位置そのものを独立に証明するものではありません。
- 位置検証には、別途の受信観測、MLAT、複数受信局での整合性確認などが必要です。
- 鍵の発行、更新、失効は必須ですが、この提案では規定していません。
- これは公開鍵暗号による電子署名方式ではありません。

## おわりに

ADS-B の本格的ななりすまし防止方式は、すでにいくつも提案されています。

しかし、それらは TESLA、PKI、Phase Overlay などを含む本格的な仕組みであり、実装、標準化、運用には大きなコストがかかります。

ここでは、もっと簡易な方式として、次の案を考えました。

```text
ADS-B に認証用フレームを 1 種類追加する
DF=17 の既存 ADS-B Extended Squitter の枠組みを使う
ME Type Code として Reserved 領域の 24 を仮に使う
ME payload は 27bit timestamp + 24bit truncated HMAC とする
既存 ADS-B フレームは変更しない
インターネット接続された受信側がオンライン検証サーバー経由で tag を検証する
HMAC 鍵は timestamp 周期より短い間隔で更新する
```

このアイディアはベーシックな暗号理論に基づいています。暗号認証の基本を知っている人であれば早い段階で思いつくような方式であり、類似の考え方は既存の ADS-B セキュリティ研究で既に議論されている可能性があります。

しかし、ADS-B のなりすましが技術的には比較的容易であることを確認した以上、リスクだけを語るのではなく、対策の方向性も合わせて示しておくべきだと考えました。
