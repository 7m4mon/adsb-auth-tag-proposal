# Simple ADS-B Authenticity Tag Proposal

Date: 11, May, 2026  
Author: 7M4MON

## 概要

これは、ADS-B のなりすまし対策について考えた、追加認証フレームの提案です。

ADS-B は航空機が自機の位置・高度・速度・識別情報などを周囲にブロードキャストする仕組みですが、現在の ADS-B メッセージ自体には暗号学的な認証機構がありません。

そのため、技術的には ADS-B らしい信号を送信する装置を作ることは比較的容易であり、悪意ある送信機によって「存在しない航空機」や「別の場所にいるように見える航空機」を作り出せてしまう可能性があります。

これは困ります。

本提案は、

> 「少なくとも、その時刻に、その正規の機体が、その空域にいた」

と言えるようにするための、追加認証フレームのアイデアです。

---

## 背景

ADS-B のなりすまし対策については、すでに本格的な提案があります。

例：

- TESLA を用いた ADS-B 認証
- PKI を用いた航空機証明書
- Phase Overlay を用いた追加データ伝送
- CABBA: Compatible Authenticated Bandwidth-efficient Broadcast protocol for ADS-B
- Secure Authentication of ADS-B Aircraft Communications using Retroactive Key Publication
- IETF draft: ADS-B Authentication

一方で、この リポジトリ で扱う案は、もっと簡易なものです。

---

## 基本アイデア

ADS-B に新しいフレーム種類を 1 つ追加し、そこに短い認証情報を載せます。

既存の位置情報フレーム、速度フレーム、Aircraft Identification フレームなどは変更しません。

追加する認証フレームの中身は、互換性を持たせるために現在のフレームと同じで以下のようにします。

```text
DF17 ADS-B Extended Squitter

| DF | CA | ICAO Address | ME Field | PI |
```

このうち、ME フィールドの Type Code に予約領域の値を仮に割り当て、認証タグ用メッセージとして扱います。

---

## 仮のフレーム割当

この提案では、DF や CA を新たに定義するのではなく、既存の 1090ES ADS-B フレームの枠組みを利用します。

仮の割当は以下とします。

| Field | Value |
|---|---:|
| DF | 17 |
| CA | 既存の transponder capability 値を使用 |
| ME Type Code | 24 |
| PI | 通常の Mode S parity |

Type Code 23〜27 は Reserved とされているため、本提案では仮に Type Code 24 を `Proposed Authenticity Tag` として使用します。

ただし、ME フィールド全体は 56 bit ですが、その先頭 5 bit は Type Code として使用されます。  
そのため、timestamp と HMAC に使える領域は 51 bit です。

本提案では以下の割当とします。

| Field | bits | 内容 |
|---|---:|---|
| Type Code | 5 | 24: Proposed Authenticity Tag |
| timestamp | 27 | 1秒単位、約4.25年周期 |
| truncated HMAC | 24 | 単発偽造成功率 1 / 16,777,216 |

```text
ME field: 56 bit

+-----------+---------------------------+------------------------+
| Type Code | timestamp                 | truncated HMAC         |
| 5 bit     | 27 bit                    | 24 bit                 |
+-----------+---------------------------+------------------------+
```

---

## HMAC の計算対象

HMAC は、機体ごとの秘密鍵を使って計算します。

例：

```text
HMAC = Truncate24(HMAC-SHA256(secret_key, message))
```

MAC 対象は、以下のとおりです。

```text
DF
CA
ICAO address
ME Type Code
timestamp
```

---

## 何を証明したいか

この方式で証明したいのは、

> その時刻に、その秘密鍵を持つ正規の送信機が、その内容のメッセージを生成した

ということです。

さらに、インターネットに接続された受信側が公開鍵情報・証明書・受信時刻と照合することで

> その飛行機がその時間にその空域にいた

という判断の信頼度を高めることができます。

---

## 受信側はインターネット接続が必要

この方式では、受信側がインターネット経由で以下の情報を取得できることを想定します。

- 機体または送信機に対応する公開鍵情報
- 証明書
- 鍵 ID
- 失効情報
- 鍵更新情報

つまり、完全にオフラインの受信機だけで完結する方式ではありません。

インターネットに接続された地上受信局や、ネットワーク接続された ADS-B 受信システムでの検証を想定します。

---

## なぜ 27bit timestamp + 24bit HMAC なのか

27bit timestamp は、1秒単位で約4.25年持ちます。

```text
2^27 seconds = 134,217,728 seconds
134,217,728 / 60 / 60 / 24 / 365 ≒ 4.25 years
```

24bit HMAC の単発偽造成功率は以下です。

```text
1 / 2^24 = 1 / 16,777,216
```

24bit HMAC は暗号学的には短いですが、この方式は本格的な認証ではなく、既存 ADS-B との後方互換性を保った なりすまし抑止 を目的としています。

また、ADS-B の場合、HMAC を総当たりするには、攻撃者は 1090 MHz で大量の偽 ADS-B フレームを送信する必要があります。

これは事実上、1090 MHz 帯を妨害しているのと近い状態になります。

そのような大量試行は、以下のような方法で検出できると思われます。

- 同一 ICAO アドレスに対する大量の認証失敗
- 異常なフレーム密度
- 複数受信局での不自然な受信状況

そのため、HMAC を長くすることも重要ですが、現実的な脅威としては、むしろ古い正規メッセージのリプレイ攻撃の方が厄介と考えます。


---

## 4.25年後のリプレイ問題と鍵更新

27bit timestamp は約4.25年で一周します。

そのため、理論上は、

> 約4.25年前に録音した正規メッセージを、同じ timestamp になるタイミングで再送する

というリプレイ攻撃が考えられます。

この対策として、timestamp 周期より短い間隔で機体ごとの鍵ペアまたは HMAC 鍵を更新します。

例えば、1年ごとの鍵更新を行えば、過去周期のメッセージを再送しても、現在の鍵では HMAC 検証に失敗します。

よって、

```text
鍵更新周期 < timestamp周期
```
とするべきです。

この観点では、24bit timestamp （約194日周期） よりも 27bit timestamp の方が扱いやすいと考えます。

---

## 鍵の漏洩と更新

この方式では、機体または送信機ごとに秘密鍵を持つことを想定します。

もし秘密鍵が盗まれた場合、その鍵を用いた HMAC は信頼できなくなります。

その場合は、以下の対応が必要です。

- 秘密鍵の失効
- 鍵ペアの更新
- 公開鍵情報の更新
- 失効情報の配布

つまり、鍵管理の仕組みは必須で、公開鍵基盤、証明書、失効リストなどを組み合わせる必要があります。

---

## 既存 ADS-B との互換性

この案では、既存の ADS-B メッセージは変更しません。

通常の ADS-B 受信機は、追加された認証フレームを理解できないかもしれませんが、既存の位置情報、速度、識別情報フレームは従来通り受信できます。

つまり、完全に後方互換性を保てます。

```text
既存受信機:
  通常の ADS-B フレームだけを見る

対応受信機:
  通常の ADS-B フレームに加えて、認証フレームも見る
  インターネット経由で署名の真正性を検証
```

---

## 「前のメッセージに HMAC をかける」案
HMACの長さを稼ぐため、

認証フレーム = 直前の ADS-B メッセージに対する HMAC

としたくなりますが、ADS-B は一方向ブロードキャストであり、ACK も再送もありません。

そのため、受信側が必ず直前のメッセージを受信できているとは限りません。

シーケンス番号を入れることも考えられますが、既存フレームとの互換性が失われます。

---

## この方式でできること

- 単純な ADS-B なりすましの抑止
- 古いリプレイ攻撃の抑止
- 正規送信機によるメッセージ生成の確認
- インターネット接続された受信側での真正性確認
- 既存 ADS-B との後方互換性維持

---

## おわりに

ADS-B の本格的ななりすまし防止方式はすでにいくつも提案されています。

しかし、それらは TESLA、PKI、Phase Overlay などを含む本格的な仕組みであり、実装・標準化・運用には大きなコストがかかります。

ここでは、もっと簡易な方式として、

```text
ADS-B に認証用フレームを 1 種類追加する
DF=17 の既存 ADS-B Extended Squitter の枠組みを使う
ME Type Code として Reserved 領域の 24 を仮に使う
ME payload は 27bit timestamp + 24bit truncated HMAC とする
既存 ADS-B フレームは変更しない
インターネット接続された受信側が公開鍵情報と照合する
鍵は timestamp 周期より短い間隔で更新する
```

という案を考えました。

既存 ADS-B との互換性を保ちつつ、単純ななりすましや古いリプレイ攻撃を抑止できる、軽量な追加認証方式として検討する価値があるのではないかと思います。
